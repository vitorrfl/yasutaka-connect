# UI conventions

- Any element that appears/disappears, or expands/collapses (drawers, dropdown/accordion sections, modals, menus), must animate both the entrance and the exit — never an instant show/hide via conditional mount. Use the "always mounted, animate via classes" pattern: keep the element in the DOM and drive opacity/transform/grid-template-rows off state, so both directions transition (see `Sidebar.tsx`'s mobile drawer and the `GroupNav` expand/collapse for the reference implementation with `grid-template-rows` for height, and translate/opacity for slide/fade).
- Any icon that represents "close" (an X) must rotate on interaction (hover and/or the action that triggers the close). Note: rotating an X by exactly 90° or 180° is invisible because the shape is symmetric under those angles — use 45° so it visibly morphs into a "+". See `Sidebar.tsx`'s mobile drawer close button for the reference implementation (separates the hover-rotate from the click-triggered close so the rotation is visible before the element fades/slides away).
- The primary button color (`Button.tsx`'s `primary` variant) is `slate-900`/`#0f172a` at rest — the same dark navy as the sidebar's own background — not a bright blue. On hover and while pressed (`:active`), it brightens to `blue-600` (`hover:bg-blue-600 active:bg-blue-600`), giving a "lights up" interaction cue. This is a deliberate brand choice: primary actions across the app should feel like an extension of the sidebar's navy identity, with the vivid blue reserved as the interactive/energized state. The sidebar's active-item accent stripe stays `blue-600` at rest (a highlight that needs to visually pop against the sidebar's own dark background — a different job than a primary button on a light page). Don't introduce different colors for new primary buttons — reuse `Button` for anything button-like so this stays automatically in sync. `blue-500`/`blue-600` remain in use for focus rings and inline links, which is fine — the "navy at rest, blue on interaction" rule only applies to button fills.
- Any search/filter field with a magnifying-glass icon (whenever a "lupa" / search is requested) MUST use the **expanding-lupa** pattern, identical to `SearchableSelect.tsx` (the reference implementation). Mechanics: at rest the lupa is a compact square (`h-10 w-10`) sitting at the right edge, with the field/content filling the space to its left. Clicking the lupa (or the field) expands the lupa's square leftward — it animates its own `width` from `w-10` → `w-full` (`transition-[width,...] duration-300 ease-out`) and *becomes* the search bar, while the adjacent field collapses its `width` to `0` with `overflow-hidden` + an opacity fade. The two widths are complementary (`field: w-[calc(100%-2.75rem)] → w-0`, `lupa: w-10 → w-full`) so they sum to 100% on every frame — no reflow, no jump, one continuous slide. The lupa icon itself is anchored `absolute right-0` inside the expanding element so it is *always* visible and never pushed out/clipped by the `overflow-hidden`. Active state turns the border/ring blue (`border-blue-500 ring-2 ring-blue-500`) and the icon `text-blue-600`; closing (lupa click / Esc / click-outside) collapses back to the square sliding right and resets the query. Both directions animate (always-mounted, animate-via-classes — same rule as above). If you need a standalone search field (not a select), extract these mechanics into a shared component so every "lupa" in the app stays automatically in sync — don't re-hand-roll the animation.
- Any destructive action (delete/excluir/apagar, remove/remover, cancel/cancelar — anything irreversible) MUST require double confirmation via the `useConfirm()` hook (`ConfirmDialog.tsx`). Never fire the action straight from the button's `onClick`, and never use the native `window.confirm`. The pattern: `const { confirm, dialog } = useConfirm();` then `onClick={async () => { if (await confirm({ title, description })) doAction(); }}`, and render `{dialog}` once in the component tree. `confirm()` resolves `true`/`false` from a `Modal`-based dialog (so it inherits the enter/exit animation and rotating X). The `description` should name the specific item and state that it can't be undone. See the delete flows in `ProductsClient.tsx` and `MateriaPrimaClient.tsx` for the reference implementation.
